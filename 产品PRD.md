产品名称：AI画原型
产品描述：AI画原型是一款基于AI技术的原型设计工具，旨在帮助用户快速生成高质量的HTML原型页面。用户只需输入设计需求，AI将根据需求生成对应的原型页面，同时提供版本控制和迭代功能，帮助用户更好地理解和修改原型。
目标用户：产品经理、设计师等需要快速创建原型的人
产品目标：通过AI对话式交互，快速生成HTML原型页面，提高原型设计效率。
优先支持：移动端原型设计。

一、 产品整体布局与交互
左侧边栏：页面/项目管理区
1.1 “+ 新页面”按钮：点击后，在列表中创建一个新的“页面X”（例如“页面13”），并在右侧激活一个新的对话和设计流程。
1.2 页面列表：展示用户创建的所有“页面X”。每个“页面X”代表一个独立的原型设计任务/想法。点击某个“页面X”可以切换到该设计任务的上下文（包括对话历史和最新原型版本）。

中间区域：原型预览区 (参考图2左侧)
2.1 固定手机外框UI元素 (中间预览区)：
- 尺寸与圆角：网页左侧（或中间，根据我们之前的布局图）的原型展示区，需要固定渲染一个类似iPhone的设备外框。
    - 外框的内容显示区域（即AI生成HTML的渲染区域）应接近 375dp (宽) x 812dp (高)（这是iPhone X/XS/11 Pro/12 mini/13 mini的逻辑分辨率）。
    - 整个外框本身会比这个内容区稍大，以包含手机的物理边框和圆角。
    - 外框需要有明显的圆角，模拟真实手机的视觉效果。
- 模拟iOS状态栏：
    - 在手机外框的顶部，固定显示一个模拟的iOS状态栏。
    - 可以包含常见元素：左侧显示时间（例如 "9:41"），右侧显示信号、Wi-Fi、电池图标。这些可以是静态图片或用HTML/CSS模拟。
- 模拟App导航栏 (底部Tab Bar)：
    - 在手机外框的底部，固定显示一个模拟的iOS底部Tab Bar。
    - 对于MVP阶段，这个Tab Bar可以是纯视觉占位，上面的图标和文字（例如“首页”、“发现”、“订单”、“我的”）是固定的，暂时不需要真实导航功能，因为AI一次只生成一个页面。
    - 它的作用是让用户在设计单个页面时，能感知到这个页面在完整App导航结构中的位置和上下文。

2.2 AI生成内容的渲染位置：
- AI生成的HTML页面内容，将被嵌入到上述手机外框UI的“内容显示区域”（即375x812dp的区域，位于模拟状态栏之下，模拟Tab Bar之上）。
- 可以使用iframe或者一个具有overflow: auto的div来承载AI生成的HTML，确保内容超出部分可以滚动，但是禁止显示滚动条。
2.3 原型展示：实时展示用户在右侧选中的原型版本（无论是A/B选项之一，还是历史版本V_X）。

右侧边栏：AI对话与控制区 (参考图1和图2右侧)
3.1 需求输入区 (初始状态类似图1中部，交互后类似图2右侧)
文本输入框：用户输入设计想法、修改意见。
图片上传按钮/区域：允许用户上传参考图片（截图、草图等），AI需要理解图片内容并作为设计参考。
示例提示 (参考图1 "试试这一个")：提供针对单个移动端页面的优质示例提示，帮助用户快速开始。例如：“设计一个App的登录页，需要有Logo区域、手机号输入、密码输入、登录按钮和忘记密码链接。”

3.2 AI交互与原型输出
AI理解与澄清：(同之前) AI理解用户文本和图片输入，进行确认和追问。
原型生成（多方案）：当用户确认需求后 (例如说“可以画原型了”)，AI会生成 2个不同的原型方案（方案A，方案B）。
方案展示与切换：在对话区以缩略图形式展示方案A和方案B。用户点击缩略图A或B，中间预览区会实时更新显示对应的原型。

3.3 版本控制与迭代
版本记录 (参考图2底部 V1, V2...)：
每当AI根据用户的（修改）需求生成新的一组原型（A/B方案）后，这代表一个新版本的诞生。
系统自动记录这些版本（V1, V2, V3...）。
用户可以点击V_X版本号，在中间预览区查看该历史版本的最终选定方案（或两个方案都可回溯，但可能更复杂，MVP阶段建议是用户选定的那个方案）。
基于特定版本迭代：
用户可以选中当前AI给出的A或B方案中的一个，或者选择一个历史版本V_X。
然后，用户在输入框中输入新的修改需求。
AI将基于用户当前选中的这个特定原型作为基础进行修改和重新生成新的A/B方案。
3.4 对话历史：(同之前) 清晰展示对话记录。
3.5 原型操作工具 (参考图1输入框下方和图2输入框下方)
查看HTML代码：提供一个按钮，可以查看当前预览区原型对应的HTML（和基础CSS）代码。
复制HTML代码：一键复制HTML代码。
全屏预览：(可能次要) 将中间预览区的原型放大到全屏查看。

二、 关键功能点强调与细化
图片输入理解：AI需要具备一定的图像识别和理解能力，能够从用户上传的图片中提取布局、元素、风格等信息作为设计参考。

移动端优先：
AI生成的HTML结构和CSS样式应首先适配移动端屏幕尺寸。
预览区使用手机外框。
示例提示也应以移动端页面为主。
双方案选择 (A/B Test for Design)：这是个很好的点！给用户选择可以提高满意度。
AI生成两个有差异化的方案。
用户选择一个更接近自己想法的，然后基于此进行迭代。
版本回溯与迭代：
清晰的“选中”状态：用户需要明确知道当前是基于哪个原型（新生成的A/B之一，或历史的V_X）在进行下一步的指令。
版本存储：每个V_X版本需要保存其对应的HTML原型代码和相关的需求描述（或至少是生成该版本的关键对话节点）。

API Key配置 (右上角设置)
入口：在页面右上角设置一个“设置”图标或按钮。
功能：点击后弹出配置界面，允许用户输入并保存不同AI服务提供商的API Key和可能的Endpoint URL。
例如：Gemini API Key, OpenRouter API Key等。
这样，每个用户可以使用自己的API配额，避免消耗你的全局配额。
安全性：API Key应存储在用户本地浏览器 (localStorage) 或通过后端安全存储（如果涉及用户账户系统，但MVP阶段前者更简单）。

三、MVP范围的再思考
核心流程：
用户创建“新页面”。
输入文字/图片描述移动端页面需求。
AI澄清，用户确认。
AI生成2个HTML原型方案 (A/B) 并在手机框内展示。
用户选择A或B，或查看历史版本V_X。
用户基于选中的原型输入修改意见。
AI基于选中原型和新意见生成新的A/B方案 (成为新版本 V_next)。
